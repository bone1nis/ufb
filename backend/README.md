## 🏁 Формат хакатона

- **5 команд**, 2 дня работы.
- Каждая команда получает **техническое задание** и реализует продукт.
- Возможные форматы продукта:
  - веб-приложение;
  - HTTP API;
  - Telegram-бот;
  - комбинированный backend с API, ботом и базой данных.
- Рекомендуемый стек:
  - **.NET** для backend;
  - **React** для frontend;
  - **PostgreSQL** или **SQLite** для хранения данных.
- В конце хакатона каждая команда демонстрирует работающий стенд с живым деплоем.

## 🌿 Ветки и git-флоу

Работаем через Merge Request.

```text
main
  └── develop
        └── feature/<task-name>
```

Рекомендуемый процесс:

1. Заберите актуальный `develop`.
2. Создайте ветку `feature/<task-name>`.
3. Реализуйте задачу.
4. Проверьте сборку локально.
5. Запушьте ветку.
6. Откройте Merge Request в `develop`.
7. После проверки команда сливает MR.
8. Pipeline на `develop` автоматически деплоит backend на стенд.

Прямые push в `main` и `develop` могут быть ограничены. Если нет прав на merge или push, обратитесь к наставнику.

## 🔁 CI/CD и деплой

Pipeline подключается из платформенного репозитория:

```yaml
include:
  - project: "hackathon-2026/platform"
    ref: "main"
    file: "workflows/backend.yml"
```

В этом репозитории описываются только переменные, специфичные для backend команды. Jobs вручную добавлять не нужно.

Pipeline выполняет:

1. проверку переменных;
2. сборку приложения;
3. сборку Docker-образа;
4. push образа в registry;
5. деплой контейнера на командный стенд;
6. healthcheck после деплоя.

После успешного pipeline backend доступен по адресу вида:

```text
https://api.team-N.hack.kam-dev.ru
```

Точный номер команды и адрес стенда сообщат наставники.

## ⚙️ Матрица сценария

Сценарий backend задаётся в `.gitlab-ci.yml` этого репозитория:

```yaml
DEPLOY_MODE: "web"
NEEDS_DB: "true"
DB_TYPE: "postgres"
```

Эти переменные должны задаваться именно в `.gitlab-ci.yml`, чтобы сценарий был виден в репозитории и одинаково воспроизводился всеми участниками команды.

| Переменная | Значения | Назначение |
|---|---|---|
| `DEPLOY_MODE` | `web`, `bot`, `full` | Режим работы backend |
| `NEEDS_DB` | `true`, `false` | Нужна ли приложению база данных |
| `DB_TYPE` | `none`, `sqlite`, `postgres` | Тип базы данных |

### Доступные сценарии

| Сценарий | DEPLOY_MODE | NEEDS_DB | DB_TYPE | Что запускается |
|---|---|---|---|---|
| API без БД | `web` | `false` | `none` | Только backend |
| API + SQLite | `web` | `true` | `sqlite` | Backend + SQLite volume |
| API + PostgreSQL | `web` | `true` | `postgres` | Backend + PostgreSQL |
| Bot без БД | `bot` | `false` | `none` | Backend в режиме бота |
| Bot + SQLite | `bot` | `true` | `sqlite` | Bot + SQLite volume |
| Bot + PostgreSQL | `bot` | `true` | `postgres` | Bot + PostgreSQL |
| Full + SQLite | `full` | `true` | `sqlite` | Расширенный runtime + SQLite |
| Full + PostgreSQL | `full` | `true` | `postgres` | Расширенный runtime + PostgreSQL |

## 🔐 Где задаются переменные

Чтобы не было конфликтов, переменные разделены по зонам ответственности.

| Где задаётся | Что хранится |
|---|---|
| `.gitlab-ci.yml` | Матрица сценария и настройки конкретного backend-репозитория |
| `env.example` | Контракт переменных и безопасные дефолты |
| GitLab CI/CD Variables | Секреты и редко меняющиеся инфраструктурные значения |

### В `.gitlab-ci.yml`

Здесь задаётся сценарий работы backend:

```yaml
DEPLOY_MODE: "web"
NEEDS_DB: "true"
DB_TYPE: "postgres"
```

Также здесь задаются технические параметры репозитория:

```yaml
SERVICE_NAME: "backend"
COMPOSE_FILE: ".platform/compose/backend.yml"
IMAGE_NAME_SUFFIX: "backend"
DOCKERFILE_PATH: "./Dockerfile"
CONTAINER_PORT: "8080"
HEALTHCHECK_PATH: "/health"
BACKEND_PORT: "8080"
```

### В GitLab CI/CD Variables

Здесь хранятся секреты:

| Переменная | Когда нужна |
|---|---|
| `DB_PASS` | Если используется PostgreSQL |
| `BOT_TOKEN` | Если используется Telegram-бот |

Секреты нельзя хранить в коде, `.gitlab-ci.yml`, `appsettings.json` или `.env` файлах в репозитории.

### В `env.example`

`env.example` не выбирает сценарий за команду. Он описывает, какие переменные существуют, какие у них дефолты и какие из них попадут в runtime `.env` контейнера.

## 🗄️ Базы данных

Backend поддерживает два варианта хранения данных.

| Тип | Когда использовать | Как работает |
|---|---|---|
| SQLite | Простые проекты, боты, прототипы | Файл БД хранится в volume backend-контейнера |
| PostgreSQL | Более сложные схемы и SQL-запросы | Отдельный PostgreSQL контейнер в сети стенда |

### SQLite

SQLite включается так:

```yaml
NEEDS_DB: "true"
DB_TYPE: "sqlite"
```

Путь к файлу БД задаётся переменной:

```env
SQLITE_PATH=/app/data/app.db
```

Файл хранится в volume, поэтому данные не пропадают при обычном пересоздании backend-контейнера.

### PostgreSQL

PostgreSQL включается так:

```yaml
NEEDS_DB: "true"
DB_TYPE: "postgres"
```

Пароль к БД задаётся через GitLab CI/CD Variables:

```text
DB_PASS
```

Параметры подключения приходят в контейнер через переменные окружения:

```env
DB_HOST
DB_PORT
DB_NAME
DB_USER
DB_PASS
```

Не прописывайте строки подключения в коде.

## 🌐 CORS и связь с frontend

> ⚠️ **Важно:** без правильно настроенного CORS frontend не сможет обращаться к backend. Браузер заблокирует все запросы с ошибкой `CORS policy`.

Backend уже содержит настройку CORS в шаблонном коде. Убедитесь, что она присутствует в вашем `Program.cs` и **не удалена** при разработке:

```csharp
// Разрешаем запросы с адреса frontend вашей команды
var frontendOrigin = GetFrontendOrigin();

builder.Services.AddCors(options =>
{
    options.AddPolicy("Frontend", policy =>
    {
        policy
            .WithOrigins(frontendOrigin)
            .AllowAnyHeader()
            .AllowAnyMethod();
    });
});

// ...

app.UseCors("Frontend");
```

Адрес frontend формируется автоматически из переменной окружения `TEAM_SLUG`:

```
https://team-N.hack.kam-dev.ru
```

Если вам нужно разрешить запросы с другого адреса (например, при локальной разработке), задайте переменную `FRONTEND_ORIGIN` в GitLab CI/CD Variables или передайте её через `.env` при локальном запуске:

```env
FRONTEND_ORIGIN=http://localhost:5173
```

### Порядок вызовов в `Program.cs`

Middleware должны вызываться строго в следующем порядке — если `UseCors` стоит после `UseRouting` или других middleware, CORS может не сработать:

```csharp
app.UseCors("Frontend");      // ← должен быть до UseAuthorization и других middleware
app.UseStaticFiles();
app.UseSwagger();
app.UseSwaggerUI();
```

### Признаки проблемы с CORS

Если frontend получает ошибки вида:

```
Access to fetch at 'https://api.team-N.hack.kam-dev.ru/...' from origin
'https://team-N.hack.kam-dev.ru' has been blocked by CORS policy
```

Проверьте:

1. Блок `AddCors` и `UseCors` присутствуют в `Program.cs`.
2. `TEAM_SLUG` корректно задан в CI/CD Variables (должен совпадать с номером вашей команды).
3. `app.UseCors("Frontend")` вызывается **до** остальных middleware.
4. После изменений выполнен push и pipeline прошёл успешно.

## 🩺 Healthcheck и служебные endpoint'ы

Приложение должно отдавать healthcheck:

```text
/health
```

Если база данных включена, могут быть доступны дополнительные endpoint'ы:

```text
/health/db
/api/db/time
```

Пример базового набора endpoint'ов:

| Endpoint | Назначение |
|---|---|
| `/` | Информация о сервисе и активном сценарии |
| `/health` | Проверка, что backend запущен |
| `/api/ping` | Простой smoke-test endpoint |
| `/health/db` | Проверка БД, если она включена |
| `/api/db/time` | Проверка запроса к БД, если она включена |
| `/swagger` | Swagger UI для API |

Список endpoint'ов может отличаться в зависимости от сценария. Например, если `NEEDS_DB=false`, endpoint'ы для БД не должны появляться.

## 🧯 Если что-то не работает

### Стенд не обновился после push

1. Откройте **CI/CD → Pipelines**.
2. Проверьте, что pipeline на нужной ветке прошёл успешно.
3. Если pipeline упал, откройте job с ошибкой.
4. Исправьте проблему и запушьте изменения снова.

### Pipeline красный, но код выглядит корректно

Обратитесь к наставнику. Возможно, проблема связана с registry, runner, deploy-host или CI/CD переменными.

### Backend запустился, но API не отвечает

Проверьте:

- есть ли `/health`;
- совпадает ли порт приложения с `CONTAINER_PORT`;
- слушает ли приложение `http://+:8080`;
- не упало ли приложение после старта;
- есть ли ошибки в логах pipeline.

### Frontend не может обратиться к backend (ошибка CORS)

Проверьте раздел [CORS и связь с frontend](#-cors-и-связь-с-frontend).

### Нужен другой сценарий деплоя

Сценарий меняется в `.gitlab-ci.yml`:

```yaml
DEPLOY_MODE: "web"
NEEDS_DB: "true"
DB_TYPE: "postgres"
```

Если для сценария нужен секрет, например `DB_PASS` или `BOT_TOKEN`, попросите наставника добавить его в GitLab CI/CD Variables.
